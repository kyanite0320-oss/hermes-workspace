## 四、Code Review 机制

### 触发时机

**每次我改完一批代码，在 git commit 之前，自动走一遍 Code Review 流水线。**

具体节点：

| 触发时机 | 内容 | 使用 skill |
|---------|------|-----------|
| 每次改完 .cs 文件（输入系统、UI 适配等） | Review C# 代码改动 | `requesting-code-review` |
| 每次改完 Unity 配置（.asset 文件） | Review 配置变更 + 语法检查 | `self-review` |
| 每次改完构建脚本（.bat） | Review 脚本逻辑 + 路径安全 | `self-review` |
| 每次 APK 构建后 | 验证构建是否成功 + 模拟器安装是否正常 | `verification-before-completion` |
| 每一阶段结束时 | 全量 Review 该阶段所有改动 | `requesting-code-review` + 飞书报告 |

### Code Review 流水线流程

```
1. git diff --cached → 出本次改动清单
   ↓
2. 静态安全检查（秘密/注入/eval/SQL 注入扫描）
   ↓
3. 编译检查 + 测试基线对比（仅 C# 相关改动）
   └─ ⚠️ Unity 项目限制：Unity Test Framework 需在 Editor 中运行，
      无法在命令行直接跑基线对比。此步骤改为 Unity MCP 在
      Play Mode 下检查编译是否通过，跳过测试基线对比环节。
   ↓
4. 独立 reviewer 子代理审核代码（无上下文偏见）
   ↓
5. 汇总结果：
    ├─ ✅ 全部通过 → 自动 git commit，飞书通知
    └─ 🔴 有问题 → 进入 auto-fix 循环（最多 2 轮）
                          ├─ 修复 → 重新走 1~4
                          ├─ 2 轮后仍未通过 → 飞书报警 + 暂停等你看
                          └─ 修复通过 → commit，飞书通知
```

### Review 内容覆盖

- 安全：硬编码密钥、Shell 注入、SQL 注入、`eval()` 滥用
- 逻辑：边界条件、空值处理、资源释放、竞态条件
- 风格：注释残留、print/debug 残留、命名一致性
- Unity 专项：
  - 资源引用的生命周期管理（防止 null ref）
  - 协程/异步代码的异常处理
  - 移动端 Input System 使用是否规范

## 五、故障处理机制

### 核心原则

> **遇到问题 → 先停、再想、再动手。**

不跳入「修修看」模式。所有非 trivial 的问题（重试 2 次仍未解决、影响后续依赖、修改涉及结构层变更）必须走固定流程。

### 故障处理流程

```
遇到问题
  ↓
第1步：判断问题等级
  ├─ 🔵 trivial（已知的配置参数调优、拼写错误、路径修正）
  │   └─ 直接修，不存档
  └─ 🟡/🔴 非 trivial（IL2CPP 兼容、插件替换、架构变更、性能瓶颈）
      ↓
第2步：加载 brainstorming skill
  ↓
第3步：分析根因
  ├─ 收集症状（报错日志、截图、Profiler 数据）
  ├─ 搞清楚是整个方案有问题还是某个参数不对
  └─ 记录关键发现
      ↓
第4步：产出问题分析文档
  ├─ 问题是什么
  ├─ 根因分析
  ├─ 可选方案（至少 2 个，含取舍）
  └─ 推荐方案 + 理由
      ↓
第5步：存入项目目录留档（C:\AI\hermes\故障记录\）
  └─ 文件名格式：YYYY-MM-DD-问题简述.md
      ↓
第6步：开始实施推荐方案
  ↓
第7步：验证 + 飞书通知结果
  ├─ ✅ 解决 → 飞书通知 + 存档标注
  └─ 🔴 未解决 → 回到第 2 步，换备选方案
```

### 示例文档结构

```markdown
# 故障记录：2026-06-XX-IL2CPP 兼容报错

> 等级：🟡 中
> 阶段：第 1 阶段 · 打通打包
> 日期：2026-06-XX

## 问题
Unity Android Build 报错：IL2CPP error ... 无法处理 System.Reflection.Emit

## 根因分析
PC 版用了 System.Reflection.Emit 的动态 IL 生成，IL2CPP 不支持运行时 JIT

## 可选方案
A. 剥离动态 IL 生成，改用静态代码生成（推荐）
B. 切回 Mono 后端（性能更差，需要测试）
C. 找 Unity IL2CPP 兼容库替代

## 推荐方案
方案 A：替换 Emit 调用为预生成的静态代码...（详细步骤）

## 结论
✅ 已解决 / ❌ 未解决，换方案
```

### 什么时候走这个流程

| 触发条件 | 级别 | 说明 |
|---------|------|------|
| 同一操作重试 2 次仍未解决 | 🟡 | 换思路，不再硬试 |
| 涉及新插件/库的引入或替换 | 🟡 | 先评估兼容性和使用方式 |
| 需要改整体架构（如输入系统方案） | 🔴 | 必须产出方案文档 |
| 性能不达标需要换渲染方案 | 🔴 | 分析瓶颈后再动手 |
| 任何需要你决策的问题 | 🟡 | 出方案后飞书喊你选择 |
| 你认为值得记录的经验教训 | 🟡 | 留档供后面参考 |

### 故障流程 vs 报警系统的关联

两个机制有以下耦合关系：

```
故障机制             报警系统
─────────────────────────────────────
重试 2 次未解决      重试 ≥3 次
  → 走 brainstorming     → 飞书 ⚠️ 报警
  → 出方案               → 通知你可能需要介入
  → 实施                  → 第 3 次还在试 = 方案没效果
  ↓                      ↓
  此时故障流程已产出方案，           若方案实施后第 3 次还失败 → 换备选方案
  报警提醒你关注
            
同一轮数的循环，故障流程先于报警触发：
  第 1 次失败 → 第 2 次失败 → 走故障流程出方案 → 方案实施 → 第 3 次失败 → 飞书报警
```

简单说：**故障流程是「怎么修」，报警系统是「让你知道在修」**，不冲突，顺序是故障先、报警后。

### 不存档的 trivial 问题

- 拼写错误、变量名错误
- 路径配置不对
- UI 锚点位置微调
- 摇杆参数 ±0.1 这种微调

#### 第 2B 部分：非像素 UI 资源生成（ComfyUI + 你确认）

> 游戏现有的非像素 UI 元素（按钮、面板、弹窗、边框、装饰等平滑风格界面组件），在手机上需要不同布局的变体。通过 ComfyUI 基于你的参考图生成，**风格保持一致**。

**我做的：**
1. 在你的 Windows 上装 ComfyUI（comfy-cli 自动化安装）
2. 搭风格迁移工作流：
   - 用你现有的非像素 UI 截图做参考
   - img2img + IP-Adapter / Reference-Only ControlNet 保持视觉风格
   - 输出 UI 元素的不同布局变体（不同比例、不同尺寸排列）
3. 按需批量生成候选图：

| 需要的资源 | 生成策略 | 数量估计 |
|-----------|---------|---------|
| 按钮变体（大/中/小、圆角/方角、不同状态） | 参考现有按钮 → style transfer → 变体 | 4~6 组 |
| 面板/弹窗变体（确认框、设置面板、商店面板） | 参考现有面板布局 → 不同尺寸变体 | 4~6 个 |
| 触屏操作条/菜单栏 | 参考现有 UI → 改布局适应触屏 | 3~4 个 |
| 装饰性 UI 元素（边框、分隔线、图标底框） | 参考图生成同风格系列 | 6~8 个 |

4. **每批生成后，通过飞书发给你确认**：候选图 + 说明
5. 根据你的反馈调整（prompt / denoise / ControlNet 权重 / 重跑）
6. 确认后 → 后处理（调色板匹配 + 裁切）→ 导入项目

**你做的：**

|------|---------|---------|
